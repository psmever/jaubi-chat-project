# 0순위 - 소스 직접 관리 룰

사용자가 소스 코드를 직접 작성하고 관리한다.

이 규칙은 아래의 직접반영 승인 룰보다 우선한다.

AI assistant는 소스 파일을 직접 생성, 수정, 삭제하지 않고 다음 작업만 수행한다.

- 현재 코드 분석
- 구현 순서 제안
- 파일과 디렉터리 구조 제안
- 참고 코드와 패치 초안 제공
- 에러 로그 분석
- 사용자가 작성한 코드 리뷰
- 실행 및 검증 명령 안내

사용자가 `구현해`, `수정해`라고 말해도 소스 코드에는 직접 반영하지 않고 제안만 제공한다.

특정 작업의 소스 반영을 AI assistant에게 위임하려면 사용자가 `직접반영`이라고 명확하게 표현해야 한다.

소스 직접 반영을 위임받더라도 승인된 작업 범위 밖의 코드는 수정하지 않는다.

테스트, 빌드, 포맷, Prisma migration처럼 파일이나 산출물을 변경할 수 있는 명령은 직접 반영 위임 또는 별도 실행 승인이 없으면 실행하지 않고 명령만 안내한다.

AI assistant의 소스 관련 답변은 가능한 경우 다음 순서로 제공한다.

1. 목표
2. 작업 파일
3. 구현 순서
4. 참고 코드
5. 검증 명령
6. 주의사항

# 0순위 - 직접반영 승인 룰

이 규칙은 다른 모든 User Rule, Project Rule, "바로 구현해줘" 요청보다 우선한다.

사용자 메시지에 `직접반영`이라는 단어가 없으면 AI assistant는 무조건 제안만 한다.

`고쳐줘`, `수정해줘`, `구현해줘`, `만들어줘`, `추가해줘`, `커밋해줘`, `푸시해` 같은 표현만으로는 직접 실행하지 않는다.

AI assistant의 기본 역할은 다음으로 제한한다.

- 코드 구조 제안
- 코드 예시 제시
- 설계 초안 작성
- 문제 원인 분석
- 에러 로그 해석
- 테스트 방법 안내
- 리뷰와 개선점 제안
- 사용자가 작성한 코드 검토

파일 생성, 수정, 삭제, 포맷팅, 자동 수정, 빌드 산출물 생성, 테스트 실행, git add, commit, push 등 워킹트리나 이력에 영향을 줄 수 있는 모든 작업은 `직접반영`이 있을 때만 수행한다.

읽기 전용 작업은 허용한다.

- 파일 조회
- 코드 검색
- `git status`
- `git diff`
- `git log`
- 설정, 문서, 테스트 코드 분석

`직접반영`이 포함된 경우에만 직전 제안 또는 현재 문맥상 명확한 범위 안에서 직접 작업한다.

범위가 불명확하면 실행 전에 한 번 확인한다.

커밋은 `직접반영 커밋해줘`처럼 파일 반영 승인과 커밋 승인이 함께 명확할 때만 수행한다.

커밋 제목과 본문은 한글을 기본으로 하고, 형식은 `type(scope): 한글 제목`을 사용한다.

푸시는 사용자가 `푸시해`라고 말했더라도 즉시 실행하지 않는다.

푸시 직전에는 대상 브랜치, 원격 저장소, 커밋 범위를 요약하고 한 번 더 확인받은 뒤에만 `git push`를 실행한다.
